一、接口概述

1.1 接口功能

getZoneTokens 返回代币分区榜单数据，支持按分区类型、时间范围、区块链网络筛选，最多返回 500 条。SEO 页面在以下场景需调用此接口：

  • Token 详情页 — 工具⑩「同类 Meme 对比」：获取同类代币做横向对比
  • Meme 购买指南页 — 相关代币推荐
  • Solana 热门代币榜单页 — 展示分区榜单（Trending / New / Up / Down 等）
  • Sitemap 生成器 — 枚举满足准入条件的代币，写入 sitemap_moonx_tokens.xml

1.2 接口参数速查

参数名类型是否必填说明
zonestring必填分区名称，见下表。固定值：trending / new / transaction / up / down / dexscreener / xstocks；或自定义分区名（如 dog、frog）
timestring必填统计时间范围：5m / 1h / 6h / 24h
networkstring可选区块链网络：eth / bsc / sol；不传则返回所有网络

二、Token 详情页 — 工具⑩「同类 Meme 对比」

程序化·改造

页面路径：/en/moonx/solana/token/{contract}

2.1 功能说明

在 Token 详情页右侧工具区第四排，展示与当前代币同类的 3 个代币，做横向对比。对比维度：24h 涨幅 / 市值 / 安全评分 / 当前价格。帮助用户判断当前代币在同类中的相对强弱。

PRD 原文（5.2.4b 工具⑩）：根据 Token 标签（dog/frog/cat/ai 等主题）自动匹配同类3个代币，横向对比：价格涨幅 / 市值 / 安全评分 / Smart Money 热度。数据来源：MoonX 内部 TokenSnapshot 表按 tag 相似度匹配。

2.2 接口调用规格

调用方式
zone = 当前代币的第一个 zone 标签（如 dog / frog / cat / ai）；若无标签则降级为 trending
time = 24h（固定，对比维度需一致）
network = sol（仅 Solana；当前 Token 详情页只处理 Solana 代币）
调用时机渲染方式刷新频率说明
页面初始加载 SSR 快照 构建时生成 Google 爬虫可读取初始同类代币数据，提升页面内容丰富度
用户浏览时 CSR 更新 每 5 分钟 实时更新涨跌幅和价格，避免数据过时

2.3 前端处理逻辑

同类代币匹配策略（按优先级降级）

优先级条件操作
1 当前代币有 zone 标签（如 dog） 调用 getZoneTokens(zone="dog", time="24h", network="sol")，从结果中排除当前代币合约，取前 3 条
2 当前代币无 zone 标签，或标签返回结果 < 3 条 降级调用 getZoneTokens(zone="trending", time="24h", network="sol")，排除当前代币，取前 3 条
3 trending 结果也不足 3 条 隐藏工具⑩整个模块，不展示空状态

展示字段映射

UI 展示项接口字段格式要求
代币 LogologoUrl圆形裁剪，24×24px；加载失败显示 symbol 首字母占位
代币名称 + 符号name + symbol如「BONK (BONK)」
24h 涨跌幅priceChange24h正数绿色 ▲，负数红色 ▼，保留1位小数，如 ▲ +12.3%
市值marketCap格式化：<1M 显示 $XXXk；>1M 显示 $X.XM；如 $1.2M
安全评分securityScore0-100 数字 + 颜色：≥70 绿色 / 40-69 橙色 / <40 红色
跳转链接contract点击整行跳转 /en/moonx/solana/token/{contract}，<a href> 内链
SEO 要求：同类代币的跳转链接必须使用 <a href="/en/moonx/solana/token/{contract}"> 标签，禁止使用 JS router.push。这样 Google 爬虫可以发现并抓取更多 Token 详情页。

五、Sitemap 生成 — Token 准入筛选

服务端定时任务

输出文件：sitemap_moonx_tokens.xml

5.1 功能说明

每日凌晨 00:00 UTC 运行一次，调用接口枚举全量代币，筛选出满足 SEO 准入条件的代币，写入 Sitemap，使 Google 发现并抓取对应的 Token 详情页。

5.2 接口调用规格

调用参数
zone = trending（覆盖最广的热门分区）
time = 24h
network = sol（仅处理 Solana，与现有 Token 详情页路径一致）
返回条数：需要 ≥ 2,000 条。Sitemap 目标 M3: 500+ URL / M6: 5,000+ URL，准入过滤后有效 Token 约为全量的 30~40%，因此原始数据需 ≥ 2,000 条才能支撑目标规模。后端需扩展接口上限（当前 100 条不够用）。
后端接口扩容需求：当前 api/web3/public/zone_tokens 返回上限为 100 条，无法满足 Sitemap 生成需求。需要后端将接口上限扩展至 ≥ 2,000 条（建议支持 limit 参数，Sitemap 任务传 limit=5000，榜单页传 limit=200，避免无谓的大数据量传输）。

5.3 准入过滤规则（PRD 5.2.4 定义）

接口返回最多 500 条，前端（服务端任务）需对结果做以下过滤，只有全部通过的代币才写入 Sitemap：

筛选条件接口字段阈值说明
LP 流动性liquidity≥ $5,000流动性过低的代币页面价值不高
持有地址数holders≥ 100持有人太少说明社区未形成
上线时间launchedAt≥ 48小时前过新的代币数据不稳定，rug 风险高
24h 交易量volume24h≥ $1,000无交易活动的代币无 SEO 价值

5.4 Sitemap 条目生成规格

每个通过准入的代币生成一条 Sitemap 记录：

<url>
  <loc>https://www.bydfi.com/en/moonx/solana/token/{contract}</loc>
  <lastmod>2026-03-19</lastmod>  <!-- 每日更新日期 -->
  <changefreq>daily</changefreq>
  <priority>0.7</priority>   <!-- 低于首页(1.0)和学院文章(0.8) -->
</url>

5.5 缓存与增量更新策略

场景处理方式
代币首次满足准入条件加入 Sitemap，同时触发 IndexNow API 推送给 Google/Bing，加速收录
代币跌出准入条件（如流动性降至 <$5K）从 Sitemap 移除；对应 Token 详情页加 noindex，但 URL 保留（不删除）
代币已 rug 或停止交易 >30天强制加 noindex，从 Sitemap 移除
Sitemap 最大 URL 数限制单个 Sitemap 文件最多 50,000 条；超过则分片为 sitemap_moonx_tokens_1.xml、sitemap_moonx_tokens_2.xml

6.2 错误处理规范

错误情况处理方式
接口返回空数组（结果为0）工具⑩/推荐模块：隐藏整个模块，不展示空状态；榜单页：显示「暂无数据，请稍后刷新」
接口超时（>3s）SSR：返回空数组，页面正常渲染，相关模块降级隐藏；CSR：保留上次数据，toast 提示「数据更新中」
接口返回 5xx 错误使用上次成功的缓存数据；同时上报错误监控，不影响页面整体加载
字段缺失（如 securityScore 为 null）安全评分展示「—」占位，不影响其他字段展示

6.3 SEO 强制要求

以下为 SEO 强制要求，验收时逐条检查：
  1. 所有通过 getZoneTokens 生成的代币跳转链接，必须使用 <a href> 标签，禁止使用 JS 跳转（onClick router.push）
  2. SSR 快照数据必须包含在 HTML 的 <body> 中，不能仅通过 JSON 注入后再客户端渲染
  3. 榜单页每个分区（Trending/New/Gainers 等）必须是独立 URL + 独立 SSR 页面，不能共用一个 URL 做客户端 Tab 切换
  4. Sitemap 中的所有 URL 必须与实际可访问页面一一对应，不能出现 404
