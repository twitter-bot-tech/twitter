## 第一部分：竞品 SEO 深度分析 — KuCoin

### 1.1 KuCoin SEO 全景

KuCoin 是目前加密领域 SEO 做得最系统的交易所之一，其打法高度工业化。

**规模数据：**

| 维度 | 数据 | 战略意义 |
|------|------|---------|
| 博客页面 | 273 页（持续更新） | 话题权威建立的内容密度 |
| 多语言 | 18 种语言 | 全球流量覆盖，每语言独立 DR 积累 |
| Sitemap 条目 | 1,000+ | 规模化 SEO 运营的标志 |
| 程序化页面 | 60+ 法币 × N 币种 = 30,000+ 页 | 长尾流量收割机 |
| 技术框架 | Next.js SSR | Google 爬取友好 |
| Domain Rating | ~75（Ahrefs 估算） | 高 DR 让新页面天然有排名优势 |

### 1.2 KuCoin 的核心武器拆解

**武器一：程序化 SEO（最强）**

KuCoin 不靠人工写作赢，靠批量生成页面赢：

```
价格页：/price/BTC, /price/ETH, /price/XRP...（500+ 页）
换算页：/converter/BTC-to-USD, /ETH-to-EUR...（3,000+ 页）
交易对：/trade/BTC-USDT, /trade/ETH-BTC...（1,000+ 页）
```

每页对应 1~3 个用户搜索意图明确的长尾词。用户搜"BTC to USD"，直接命中。这是人工写作永远追不上的规模。

**武器二：内容集群 + 内链转化链路**

```
市场洞察文章（流量入口）
    ↓ 内链
价格预测页（承接流量）
    ↓ 内链
实时价格页 /price/[token]（数据展示）
    ↓ 内链
交易入口（转化终点）
```

每层都有内链，流量不浪费，权重向转化页汇聚。

**武器三：多语言规模化**

18 种语言意味着同一篇内容产生 18 倍的 SEO 覆盖。对高 DR 站点是乘法效应。

### 1.3 KuCoin 的弱点（MoonX 的机会）

| 弱点 | 对 MoonX 的机会 |
|------|--------------|
| 预测市场赛道完全空白 | MoonX 可成为该赛道 Google 唯一权威 |
| 内容同质化（大量价格预测文章） | Google HCU 持续降权此类内容 |
| 品牌权重依赖 | 新站在细分赛道反而更灵活 |
| 多语言稀释了内容深度 | 专注单语言精品内容更高效 |

### 1.4 核心判断

**KuCoin 打的是交易所战争，MoonX 打的是预测市场聚合战争。**

这两场战争的战场在 Google 上几乎不重叠。MoonX 不需要赢过 KuCoin，只需要在预测市场赛道成为 Google 眼中的第一权威。

## 第三部分：MoonX SEO 完整战略

### 3.1 战略定位

> **MoonX 的 SEO 战略：成为 Google 眼中"预测市场聚合"赛道的唯一权威。**

不追 KuCoin 的规模，不打交易所关键词，专注预测市场这个细分赛道，用 6 个月时间建立话题权威（Topical Authority），让 Google 在所有预测市场相关查询中优先信任 MoonX 的内容。

### 3.2 三层 SEO 架构

```
第一层：内容权威（Topical Authority）
  → Pillar Page + Cluster Pages 内容集群
  → 覆盖预测市场所有搜索意图

第二层：程序化规模（Programmatic SEO）
  → 平台聚合页 + 话题页批量生成

第三层：数据驱动优化（Data Intelligence）
  → GSC + GA4 + Twitter 数据管道
  → Claude Code 跨源分析，持续优化
```

## 第五部分：SEO 数据体系搭建（Claude Code 指挥中心）

参考文章架构，结合 MoonX 实际需求，定制版数据管道：

### 5.1 项目结构

```
moonx-seo/
├── config.json                    # MoonX 配置文件
├── fetchers/
│   ├── fetch_gsc.py               # GSC：关键词排名、展示量、点击率
│   ├── fetch_ga4.py               # GA4：流量来源、页面表现、转化
│   └── fetch_twitter.py           # Twitter：推文热度、话题趋势（MoonX特有）
├── data/
│   ├── gsc/                       # 关键词数据 JSON
│   ├── ga4/                       # 流量数据 JSON
│   └── twitter/                   # 推文数据 JSON
├── reports/                       # Claude Code 生成的分析报告
└── run_fetch.py                   # 一键拉取所有数据
```

### 5.2 config.json 配置

```json
{
  "name": "MoonX - BYDFi Prediction Market Aggregator",
  "domain": "bydfi.com",
  "moonx_path": "bydfi.com/en/moonx",
  "blog_path": "bydfi.com/en/blog",
  "gsc_property": "https://www.bydfi.com/",
  "ga4_property_id": "[待填写]",
  "industry": "Prediction Market / Crypto",
  "target_keywords": [
    "polymarket alternative",
    "prediction market aggregator",
    "crypto prediction market",
    "kalshi vs polymarket",
    "polymarket smart money"
  ],
  "competitors": [
    "https://polymarket.com/",
    "https://kalshi.com/",
    "https://manifold.markets/"
  ]
}
```

### 5.3 搭建步骤（按优先级）

**Step 1：GSC 接入（本周）**
```
1. Google Cloud Console 创建项目
2. 启用 Search Console API
3. 创建服务账户，下载 JSON 密钥
4. 将服务账户邮箱加入 GSC 属性（只读权限）
5. 运行 fetch_gsc.py 验证数据通路
```

**Step 2：GA4 接入（本周）**
```
1. 确认 GA4 tracking code 已装在 BYDFi 博客
2. 启用 Google Analytics Data API
3. 将服务账户加入 GA4 属性（查看者权限）
4. 运行 fetch_ga4.py 验证数据通路
```

**Step 3：Twitter 数据接入（第二周）**
```
1. 复用现有 tweet_bot.py 的 Twitter API 认证
2. 写 fetch_twitter.py：拉取推文表现 + 话题热度
3. 与 GSC 数据交叉：哪些 Twitter 话题对应哪些 GSC 关键词
```

**Step 4：跨源分析问题库（持续补充）**

向 Claude Code 提问的高价值问题：

```
GSC × GA4：
"哪些页面 GSC 展示量高但点击率低（CTR < 2%）？
 对应的 GA4 跳出率是多少？优化标题还是内容？"

GSC × Twitter：
"过去 7 天 Twitter 上预测市场热门话题是什么？
 对应的 GSC 关键词排名如何？有没有内容缺口？"

内容缺口分析：
"对比我们已有文章的关键词，
 找出月搜索量 >500 但我们完全没有内容的词。"

竞品差距分析：
"Polymarket.com 和 Kalshi.com 在哪些关键词上排名前 10 但我们完全缺席？"
```

| 数据报告 | 发布独家市场数据报告，媒体主动引用 | 高质量外链爆发 |
